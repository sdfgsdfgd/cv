# CV Todo

## ATS-safe version context, deferred

Do not do this in the current pass. Current priority is the beautiful, hand-designed CV.

Why keep an ATS-safe version on the roadmap anyway:

- LinkedIn Easy Apply still sends an uploaded resume to the job poster, and some job flows route applicants into external applicant tracking systems.
- LinkedIn Recruiter uses skills, location, profile text, resume skills, keywords and filters. Even if a human ultimately decides, machine-readable wording affects whether the human sees the profile early.
- Recruiters and agencies often export resumes into their own systems. A beautiful two-column PDF can be excellent for a hiring manager, but less robust for parsing, searching and database reuse.
- This should not replace the beautiful CV. The useful setup is two artifacts:
  - beautiful PDF for humans, referrals, direct messages and sophisticated hiring teams;
  - clean one-column ATS-safe PDF/DOCX for Easy Apply, portals, agencies and recruiter databases.

What the ATS-safe version should eventually do:

- One-column layout, normal headings, standard fonts, no letter-spaced headings, no ornaments.
- Same strongest story as the beautiful version: senior Android/Kotlin engineer for payments, wallets, security-sensitive systems and high-trust delivery.
- Dense but natural keyword coverage: Kotlin, Jetpack Compose, Kotlin Multiplatform, Compose Multiplatform, Coroutines, Flow, MVVM, MVI, Clean Architecture, modularisation, Room, Dagger/Hilt, Retrofit/OkHttp, GraphQL, gRPC, REST, WebSockets, Firebase, AWS, CI/CD, GitHub Actions, Jenkins, Fastlane, Maestro, unit tests, instrumented tests, PCI, PayWave, Tap-to-Pay, EFTPOS, secure key handling, encryption, wallets, Play Store/release ownership, observability.
- Year-only dates if that remains the chosen strategy, but exact chronology should be kept in a private source-of-truth table.
